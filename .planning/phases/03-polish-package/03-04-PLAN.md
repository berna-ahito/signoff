    1. # ProcureFlow AI — Case Study

    2. ## The Problem — 2-3 paragraphs:
       - Purchase requests in organizations are typically managed via email chains or informal spreadsheets, creating inconsistent approval workflows, missing audit trails, and no standardized risk assessment
       - The challenge: demonstrate an AI-assisted intake system that stays in bounds (AI recommends, humans decide) while being realistic and portfolio-grade
       - Target audience note: this is a portfolio piece demonstrating production-grade patterns on a realistic domain

    3. ## Key Design Decisions — 5 decisions in subsections:

       ### AI Boundary Enforcement (Human-in-the-Loop)
       - Decision: AI may classify, summarize, assess risk, and draft RFQ text. AI may NOT approve spending.
       - Rationale: Prevents liability, maintains human accountability, demonstrates responsible AI integration
       - Implementation: recommended_action is stored as advisory text; the approver's decision is a separate recorded field

       ### Provider-Agnostic AI Layer
       - Decision: AIReviewProvider ABC with MockProvider default; GroqProvider available via config
       - Rationale: No paid API dependency for development/testing; real providers swap in with zero business logic change
       - Trade-off: Mock output is deterministic, not truly intelligent — adequate for demonstrating the architecture

       ### Configurable Approval Rules Engine
       - Decision: approval_rules DB table evaluated at submit time; first-match rule determines assigned_role
       - Rationale: Avoids hardcoded thresholds (which break when policies change); rules are testable and auditable
       - Implementation: rule.min_amount, rule.max_amount, rule.category (nullable for wildcard), rule.required_role, rule.priority

       ### Security: IDOR + Rate Limiting + No Mass Assignment
       - Decision: _get_request_or_403 helper for ownership checks; SlowAPI for per-IP rate limiting; separate Create/Update/AdminUpdate schemas
       - Rationale: OWASP Top 10 compliance for a real-world procurement system that handles financial data
       - Trade-off: Separate schema classes add boilerplate but prevent entire classes of injection vulnerabilities

       ### SQLite Local-First
       - Decision: SQLite via SQLAlchemy for local development
       - Rationale: Zero infrastructure for portfolio demonstration; Alembic migrations mean a swap to PostgreSQL requires only a connection string change
       - Trade-off: No concurrent write support; acceptable for single-developer portfolio use

    4. ## Security Measures — bulleted list (6-8 items): IDOR guard, rate limiting details, JWT payload, no mass assignment, audit log, bcrypt, .env.example only, status transition validation

    5. ## Trade-offs Made — table with Decision / Trade-off / Why Accepted columns:
       - MockProvider / Not real AI output / Free, deterministic, demonstrates the architecture correctly
       - SQLite / No concurrent writes / Zero infra, easy demo
       - localStorage JWT / Vulnerable to XSS if CSP not set / Acceptable for portfolio; production would use httpOnly cookies
       - No full integration test suite / Less coverage than ideal / 79 unit tests + 16+ frontend tests sufficient for portfolio; httpx E2E tests deferred
